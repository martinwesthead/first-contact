## Story

**As a** small-business site owner whose site is hosted on the 1st Contact platform,
**I want** to open the builder at `app.1stcontact.io/builder` and edit my live site definition by chatting in plain English with an AI assistant on the left while watching my site change in a live preview on the right,
**so that** I can adjust copy, visual settings, sections, and theme without learning a CMS, writing code, or losing track of which change I made.

**As an** operator using the builder day-to-day,
**I want** a two-panel layout in which the chat side collapses to a thin restore rail on the left edge of the preview, the splitter between the two panels can be dragged to resize them, the preview supports mobile / tablet / desktop viewport presets, and all of these layout choices plus my working site definition survive a page reload,
**so that** the builder fits my screen the way I like it and I never lose in-progress edits when I close the tab.

**As the** 1st Contact platform operator,
**I want** every AI tool call to be checked against the framework's module catalog and the site schema before it is applied to my state, and rejected tool calls to be surfaced as a structured error in the chat log without changing the underlying site,
**so that** the AI cannot corrupt my site definition and I can see exactly what it tried and why it was refused.

**As the** 1st Contact platform operator,
**I want** the AI tool list available in any chat session to reflect exactly what my plan tier permits me to do, so that an unauthenticated or trial session sees only the state-edit tools while a paid session additionally sees system-action tools (e.g. `publish_stub`),
**so that** the AI never offers an action my plan does not authorise and the platform's "Operator API parity" principle is enforced uniformly at the tool-surface boundary.

**As the** 1st Contact platform operator,
**I want** the AI to receive a structured result after every tool call and to be able to read the current draft definition back on demand, the assistant's reply to stream into the chat token-by-token as it is generated, both my own messages and the assistant's replies rendered as formatted markdown (not raw text), a chat input that understands pasted markdown, a collapsible tool-use pane that shows what the AI is doing live, structured tool outcomes shown as consistent cards in the chat log, and a Send button that turns into a Stop button I can click to abort the turn while it is still running,
**so that** the AI reasons from confirmed state rather than its own inference across a multi-step change, the chat is legible, visually consistent, and feels responsive, and I stay in control of a long-running turn.

## Description

This story documents the Phase 0 chat-driven site builder: the operator-facing single-page application that turns plain-English nudges into structured edits against a `Site` definition, validated against the framework catalog and `@1stcontact/site-schema` before being applied, with a live in-browser preview that re-renders on every accepted change. It also covers the `/builder` route on `apps/control-app` (served from Workers Static Assets) and the `POST /api/chat` endpoint that proxies the Anthropic Messages API with a plan-tier-filtered tool surface derived from the `OPERATOR_ACTIONS` registry, the framework catalog as a system prompt, and structured extraction of text + tool_use blocks. `/api/chat` consumes Anthropic's upstream streaming response and **re-emits the turn to the browser as a Server-Sent Events stream** (`token` / `tool_call` / `tool_result` / `done` / `error` events) rather than a single JSON block. The endpoint runs a multi-turn tool loop: after each Anthropic response carrying tool calls it executes each call server-side, feeds a structured `tool_result` back to the model, and recomputes the site-state snapshot in the system prompt from the canonical working definition before the next turn — capped at 8 turns. The client driver grows an in-flight assistant bubble token-by-token as `token` events arrive. Both assistant and user replies render as sanitized markdown, the chat input is a rich-text (TipTap) rounded-pill editor with markdown round-trip and a placeholder hint, a collapsible tool-use pane surfaces live tool events above the message list, and structured tool outcomes render through a reusable card primitive.

In scope:

- A two-panel SPA shell with a chat panel on the left, an iframe preview on the right, and a draggable splitter between them.
- The chat panel collapses to a 32px restore rail that appears on the **left** edge of the preview (where the chat was), and restores to its remembered width on click. Both panel width and collapsed/expanded state are persisted across reload.
- The preview iframe fills its panel's height (does not collapse to a default 150px), and supports three viewport presets — mobile (375px), tablet (768px), and desktop (100% of the pane).
- The preview supports **in-document multi-page navigation**. The framework renderer takes a `target: 'preview' | 'production'` option (default `'production'`, production output byte-unchanged); in `'preview'` mode it emits page-nav links as fragment hrefs (`#/<pageId>`) instead of absolute paths. The preview driver renders the iframe with `target: 'preview'` and installs a `hashchange` listener on the iframe's content window, so clicking a page tab re-renders the matched page's modules in place — via `document.open`/`write`/`close`, with no HTTP round-trip — instead of navigating the iframe away to a control-app URL. `#/` returns to the home page, in-page anchor hashes (e.g. `#contact`) do not switch pages, and an unknown `pageId` falls back to the first page.
- The preview panel toolbar carries a **Reset** button (right-aligned, after the viewport-preset buttons). Clicking it asks for confirmation ("Reset the site to the 1stcontact baseline? Edits will be lost."); on confirm it clears the persisted working-site key from localStorage and reloads, so the builder cold-loads the bundled baseline starter site; on cancel nothing changes. This is a dev affordance for iterating on the convert flow without hand-clearing localStorage.
- The eight state-edit AI tools — `set_module_content`, `set_module_dial`, `set_module_variant`, `add_module`, `remove_module`, `reorder_modules`, `set_theme_token`, `set_site_config` — live in the `OPERATOR_ACTIONS` registry with `category: 'state_edit'`, `plan_tier: 'trial'`, and `ui_route: null` (chat-driven only). `POST /api/chat` constructs the Anthropic tool list by filtering this registry through the session's `plan_tier`, so a trial-plan session receives the eight state-edit tools while a paid-plan session additionally receives paid-tier system-action tools such as `publish_stub`. There is no separate static `TOOL_DEFINITIONS` constant: the registry is the single source of truth.
- Four-layer validation contract: every AI-returned tool call is checked against the framework catalog (module type, version, variants, dials, theme-token names) and the site schema before being applied to the store. Accepted tool calls move the working site forward; rejected tool calls leave state unchanged and append a structured error message (path, expected enum, got, message) to the chat log.
- localStorage persistence of the working site definition and the chat-turn history; on reboot, a previously-edited site survives and chat history is preserved across store re-instantiation.
- A `POST /api/chat` endpoint on `apps/control-app` that proxies the Anthropic Messages API using the bound `CLAUDE_API_KEY` secret (default model `claude-sonnet-4-6`, overrideable via `CLAUDE_MODEL`), with the plan-tier-filtered tool list (see above) and the framework catalog as the system prompt. The endpoint requests Anthropic with `stream: true` and returns a `text/event-stream` response, parsing the upstream `content_block_start/delta/stop` frames and re-emitting `token` (text delta), `tool_call` (a reassembled tool_use block), `tool_result` (server-side execution result), `done` (final `{text, toolCalls, systemActions, intentToken}` aggregate), and `error` events to the browser. Responds 500 when the API key is unbound and 502 when the upstream call errors or returns a non-2xx response.
- `apps/control-app` routes `/builder` and `/builder/` to the Workers Static Assets binding by rewriting to `/builder.html`; all other paths fall through to the static assets binding first, then the control-app placeholder.
- A starter site (`/starter-sites/1stcontact.json`) is fetched at SPA boot from the same origin; the `?site=` query param selects which starter to load (defaults to `1stcontact`).
- Multi-turn AI tool loop on `/api/chat`: after each Anthropic response that returns tool calls, the worker executes each call, appends a structured `tool_result` content block (`{ok: true, applied: {tool, args, summary}}` on success — `data`/`kind` added when the tool returns structured payload — or `{ok: false, error: {tool, validation}}` on failure, with the failure also flagged `is_error` on the block returned to Anthropic), recomputes the system-prompt site-state snapshot from the canonical working definition, and re-calls Anthropic until no tool calls remain or 8 turns are reached.
- A `get_site_definition` read tool in `OPERATOR_ACTIONS` (`category: 'system_action'`, `plan_tier: 'trial'`, `ui_route: null`) whose handler returns the current draft site definition so the AI can verify canonical state mid-conversation; it is available on every plan tier and its returned payload is surfaced to the model as `applied.data`.
- **End-to-end streaming**: the client chat driver appends an empty assistant bubble at turn start and grows it as each `token` SSE event arrives, so the reply renders progressively rather than appearing as a single block on completion; a stream-level `error` event surfaces a sorry-message bubble in place of the in-flight one.
- **Send / Stop swap**: a round accent Send button (play glyph) sits inside the input capsule; while a turn is in flight it is hidden and a round red Stop button (square glyph) is shown in its place. Clicking Stop aborts the turn via a per-turn `AbortController` (which also cancels the SSE reader and the upstream Anthropic request). Repeat sends during an in-flight turn fire no second send and an empty input no-ops first; the editor stays editable throughout. The old CSS spinner overlay is removed.
- **Collapsible tool-use pane**: a collapsible pane above the message list (chevron header that flips ▶ ↔ ▼) shows live tool events for the current turn — it expands and appends a row as each `tool_call` arrives and is reset at the start of each new turn. Inline `ChatCard` tool-result renderers in the message list are retained alongside the pane (the pane is the live "what is the AI doing now" view; inline cards are the durable result artefacts).
- Both assistant and user chat messages render as markdown (headers, lists, fenced code become DOM elements; links open in a new tab with `rel="noopener noreferrer"`) sanitized so injected `<script>` tags and `on*=` event-handler attributes are stripped; only system messages remain plaintext.
- The chat input is a rounded-pill rich-text editor (13px typography, markdown typography for pasted content, and an "Enter to send, Shift+Enter for newline" placeholder hint) that accepts pasted markdown and shows it as formatted text, and serializes its content back to markdown for the `/api/chat` request. **Bare Enter sends the message; Enter with any modifier (Shift / Alt / Meta / Ctrl) inserts a newline.** The Send/Stop button is positioned inside the capsule at its right edge.
- A reusable chat-card primitive (bordered card with an icon+title header, body slot, optional actions row with click callbacks, optional collapse caret, and one of five tones — neutral / info / success / warning / danger — applying an edge accent) plus a `tool_result` kind→renderer dispatcher: known kinds route to a registered renderer, unknown or untagged successful results fall back to a markdown card showing the summary, and failed results render as a danger-toned card. Downstream REQs register their per-kind renderers through this dispatcher.

Out of scope (deferred per the source ticket):

- D1 save / load of edited site definitions.
- Auth gating on `/builder`; the route is open in this slice.
- The deferred half of the DOC-8 §5.1 tool surface: `set_nav_pattern`, `set_nav_entries`, `attach_asset`, `add_page`, `remove_page`, `reorder_pages`.
- Diff visualisation, click-to-edit in preview, drag-to-reorder, per-module diff rendering (page-level re-render is current granularity), and mobile-friendly builder UI.

## Technical Context

- **Validation contract** is shared with **CAP-32 / Site Definition Schema** (`validateSite`) and **CAP-34 / Framework Module Catalog** (module metas exposed via the browser-safe `@1stcontact/framework/meta` subpath). The builder catalog (`buildFrameworkCatalog()`) compresses the framework metas into a `{modules, themeTokenNames}` shape the chat handler and tool validator can both consume cheaply.
- **In-browser render path**: the preview iframe is filled by `renderSiteToHtml(site)` from `@1stcontact/framework/render` (in-browser variant, no Node-fs runtime imports). The static site generator (CAP-35) and the in-browser preview share the same framework module catalog but use different render entry points.
- **Preview multi-page navigation (BUG-3)**: the framework renderer is shared between the production Astro static-site generator (one HTML file per page) and the in-browser preview (a single document). A `RenderTarget = 'production' | 'preview'` option on `RenderSiteOptions` (default `'production'`) threads through `renderHeader` to `navHref`, which emits `#/<pageId>` for `kind: 'page'` nav entries in preview mode and `/<pageId>` in production mode — `kind: 'url'` and `kind: 'anchor'` hrefs are identical across targets, so production output is byte-unchanged. `renderSiteIntoIframe` (`packages/builder-ui/src/preview.ts`) tracks per-iframe state in a `WeakMap`, resolves the active `pageId` from the iframe's hash, renders with `target: 'preview'`, and installs a one-shot `hashchange` listener on the iframe's `contentWindow`. `pageIdFromHash` matches only `#/<pageId>` fragments (`#/` ⇒ home / first page); bare in-page anchors like `#contact` return `undefined` and leave the current page untouched, and an unrecognised `pageId` falls back to `site.pages[0]`. Re-render is in-document (`document.open`/`write`/`close`), so page switching makes no HTTP request and never reaches the control-app catch-all. Fragment-based nav (DOC-8 §3.2 Shape 1) was chosen over click-interception because it survives the iframe being reloaded by other means.
- **Anthropic API**: the proxy uses `https://api.anthropic.com/v1/messages` by default (overrideable via `ANTHROPIC_API_URL` for testing), the `anthropic-version: 2023-06-01` header, and the API key from the `CLAUDE_API_KEY` secret. The system prompt embeds the catalog and the first 16K characters of the current site definition; messages are filtered to `user`/`assistant` roles only.
- **End-to-end streaming (REQ-36)**: `apps/control-app/src/chat.ts` issues the upstream Anthropic request with `stream: true` and returns a `Response(ReadableStream, content-type: text/event-stream)`. `consumeAnthropicStream()` parses the upstream `content_block_start` / `input_json_delta` / `content_block_stop` frames, accumulating `partial_json` per `tool_use` block, and the handler emits SSE frames to the browser: `token { delta }` (forwarded text deltas), `tool_call { name, input, toolUseId }` (the moment a tool_use block's input JSON is fully reassembled), `tool_result { name, input, result, toolUseId }` (server-side execution result), `done { text, toolCalls, systemActions, intentToken }` (final aggregate), and `error { error }` (fatal before/during the stream). On the client, `packages/builder-ui/src/chat-driver.ts` replaces the single-fetch JSON read with `response.body.getReader()` + an SSE frame parser: it appends an empty in-flight assistant bubble at turn start, grows it via `store.updateLastChatMessage()` on each `token`, fires `onToolCallStart`/`onToolCallResolved` for the tool-use pane, and commits the final `text` + structured `toolCalls` on `done`; a stream `error` (or a non-2xx response) replaces the bubble with a `Sorry — …` message. This supersedes the earlier "narrative text only renders on turn complete" deferral.
- **Send/Stop swap + abort (REQ-36)**: `chat-panel.ts` renders a round `.fc-chat__send` (accent, ▶ glyph) and a round `.fc-chat__stop` (red, ■ glyph), both absolute-positioned inside the editor capsule. While busy, `data-fc-chat-send-busy` hides Send and `data-fc-chat-stop-visible` reveals Stop; the `ChatPanelHandle` exposes `stopButton`, and the chat driver wires its click to the per-turn `AbortController` so Stop aborts the SSE reader and the upstream request. The REQ-32 CSS spinner overlay is removed; the legacy REQ-32 Cmd+Enter and spinner tests were updated to drive submission via the Send button click and to assert the Send/Stop swap.
- **Keyboard contract (REQ-36)**: `chat-panel.ts` mounts TipTap with `editorProps.handleKeyDown` — a bare Enter submits, and Enter with any of Shift/Alt/Meta/Ctrl falls through to TipTap's default newline insertion. This replaces the prior REQ-13 Cmd/Ctrl+Enter-to-send contract; the placeholder advertises "Enter to send, Shift+Enter for newline".
- **Tool-use pane (REQ-36)**: `chat-panel.ts` adds `.fc-chat__tool-header` (chevron + label) and a collapsed-by-default `.fc-chat__tool-pane` with a `.fc-chat__tool-pane-body`. The header toggles the pane (chevron ▶ ↔ ▼). The panel exposes `appendToolEvent(detail)`, `clearToolEvents()`, and `expandToolPane()`; `main.ts` wires the driver's `onToolCallStart`/`onToolCallResolved` callbacks to append rows and expand the pane. Inline `ChatCard` result renderers in the message list are kept alongside.
- **User-message markdown (REQ-36)**: `renderMessage` now routes both `user` and `assistant` roles through `renderMarkdownToDom` (system messages stay plaintext via `textContent`), so an operator's typed markdown renders formatted in its bubble. This widens the REQ-13 contract, which markdown-rendered only assistant messages.
- **Chat capsule styling (REQ-36, supersedes BUG-1)**: `apps/control-app/public/builder.html` overhauls the input CSS — `.fc-chat` sets `font-size: 13px; line-height: 1.6`; `.fc-chat__editor` is a `border-radius: 24px` single-border capsule with a `#2563eb` focus accent, `min-height: 44px`, `max-height: 232px`; `.fc-chat__editor-content` uses `padding: 10px 62px 10px 14px` to clear the inline button and styles `h1/h2/h3`, lists, blockquote, pre/code, and table typography for pasted markdown. The placeholder renders via `@tiptap/extension-placeholder`. The REQ-13/BUG-1 `.fc-chat__textarea` and REQ-32 `.fc-chat__send-spinner` rules are gone.
- **OPERATOR_ACTIONS registry as single source of truth**: the eight state-edit tools previously declared as a static `TOOL_DEFINITIONS` constant in `chat.ts` are gone; they live alongside system actions in `OPERATOR_ACTIONS` and are dispatched through the same registry. State-edit tools retain their REQ-8 client-side validator + application semantics — moving them into the registry is a tool-list-construction change, not an execution-path change.
- **Plan-tier-derived tool list**: the chat handler extracts a session record (`{session_id, account_id, plan_tier}`) from the request headers (`x-session-id`, `x-account-id`, `x-plan-tier`), defaulting to `{anonymous, trial}` when headers are absent. The Anthropic `tools` array is then computed by `visibleToolSpecs(plan_tier)`, which filters `OPERATOR_ACTIONS` by `tierPermits(session.plan_tier, action.plan_tier)`. Trial sessions see all `plan_tier: 'trial'` actions; paid sessions additionally see `plan_tier: 'paid'` actions; enterprise additionally sees `plan_tier: 'enterprise'` actions.
- **SPA bundle**: `apps/control-app/scripts/build-builder-bundle.mjs` runs esbuild to produce `apps/control-app/public/_assets/builder.js` from `packages/builder-ui/src/spa.ts`. Both the bundle output and the bundled starter site copy under `public/starter-sites/` are gitignored — they rebuild deterministically from the `build`/`deploy`/`dryrun` scripts.
- **Framework-choice divergence from the source ticket**: the source ticket named React components; the implemented slice uses vanilla DOM (`BuilderStore` + DOM factory functions). The user-visible behaviours documented in this story — panel/splitter/viewport UX, tool validation, preview re-render, persistence, the chat endpoint — are framework-agnostic and continue to hold after a hypothetical React rewrite. The matrix documents what the user sees, not the framework choice.
- **DOC-8 §5.3 invalid-dial example**: DOC-8 names `shape: 'cirle'` as the canonical invalid-dial UAT. None of the Phase 0 modules declare a `shape` dial, so the implemented UAT asserts on `size: 'huge'` against the hero (whose `size` dial is `[sm, md, lg]`). Same validator code path, same DOC-8 §6 invariant pinned.
- **Tool validation pre-check note**: the implemented tool validator does NOT separately catalog-validate the `add_module`'s `variant`/`dials` — those land via `validateSite`'s structural rules. The behaviour the user observes (invalid values are rejected, state is unchanged, chat log shows a structured error) is unchanged regardless of which validator layer surfaces the rejection.
- **Multi-turn loop (REQ-13)**: the chat handler loops up to `MAX_TOOL_TURNS = 8`. State-edit calls advance an in-handler working copy of the site (`applyToolCall`) and the success `summary` describes what landed (e.g. `"set dial 'spacingTop' to 'lg' on hero-1"`); the system prompt is rebuilt each turn from that working copy, so continuation accuracy holds across long sessions. `get_site_definition` is dispatched as a system action and its payload is surfaced to the AI via the legacy no-`kind` path (`applied.data`), while any system action returning a `payload.kind` string surfaces `applied.data` + `applied.kind` on the `done` payload's `toolCalls` for the FE dispatcher.
- **Markdown rendering (REQ-13)**: assistant (and, per REQ-36, user) messages go through `marked` (v14, GFM on, a custom link renderer adding `target="_blank" rel="noopener noreferrer"`) then `DOMPurify.sanitize`; system messages use `textContent`. The chat input is a TipTap editor (`StarterKit` + `tiptap-markdown` with `html: false`), serializing back to markdown on submit.
- **Preview-panel Reset button (REQ-31)**: `createPreviewPanel` renders a Reset button in the toolbar only when an `onReset` callback is supplied; `bootBuilder` wires it to confirm via `window.confirm`, then `localStorage.removeItem('1stcontact_builder_site_v1')` and `location.reload()`. The confirm prompt, the reload primitive, and the storage key are injectable (`resetPrompt`, `reloadPage`, `storageKey`) so UATs exercise the flow without reloading jsdom. A `window.confirm` prompt (not a styled chat-card modal) is intentional for this dev affordance.
- **ChatCard + dispatcher (REQ-13)**: implemented as the vanilla-DOM `createChatCard` factory and a `kind → renderer` registry (`registerToolResultRenderer` / `renderToolResult`), not the React `<ChatCard>`/`react-markdown` the source ticket named. The user-visible behaviours (card shell + tones + collapse, markdown fallback, danger card on failure, per-kind routing) are framework-agnostic — the matrix documents what the user sees, not the framework choice (consistent with the divergence note above).

## Dependencies

- **CAP-32 / Site Definition Schema** — `validateSite()` is the schema-validation half of the four-layer contract; `Site`/`ModuleInstance`/`Page` types are the canonical site-definition shape.
- **CAP-33 / Framework Theme Tokens & CSS Generation** — the in-browser preview consumes generated theme CSS so style edits visually land in the iframe.
- **CAP-34 / Framework Module Catalog** — the chrome and content modules are what the catalog enumerates and what the preview renders; the `@1stcontact/framework/meta` browser-safe subpath is the bundle bridge.
- **Operator API (REQ-9)** — the `OPERATOR_ACTIONS` registry, the plan-tier authorization middleware, and the `tierPermits` ordering that filters the tool list before each Anthropic request.

## Story

**As a** visitor to a 1st Contact-built marketing site,
**I want** to submit the contact form and receive immediate confirmation that my inquiry was received,
**so that** I trust the business will see my request and respond.

**As a** small-business owner whose site is served by the 1st Contact platform,
**I want** every legitimate contact-form submission to be reliably recorded as a lead and to notify me by email, while obvious spam is silently dropped and automated abuse is challenged by Turnstile,
**so that** I never lose a real customer inquiry to noise and never get woken up by bot traffic.

**As the** 1st Contact platform operator,
**I want** the lead store shaped from day one for the full CRM Lite lifecycle (new → contacted → quote_needed → quote_sent → follow_up → booked / lost / archived), and for each lead row to carry enough provenance (site_id, form_id, page_path, user_agent, ip_country, turnstile_pass, extra_fields JSON) to support downstream triage, audit, and per-site routing,
**so that** the CRM Lite UI and per-site lead routing can be added later without a data migration.

## Description

This story documents the end-to-end lead-capture pipeline that replaces the earlier stub contact endpoint: the D1 `leads` schema, the `POST /api/forms/contact` handler on the public-site Worker, Cloudflare Turnstile verification, best-effort Resend email notification, the Turnstile script-tag injection performed by the static site generator when a page contains a contact-form module, and the contact-form client island's behaviour of attaching the Turnstile token to its JSON submission.

In scope:

- A `leads` table created by `db/migrations/0001_create_leads.sql` containing every column the CRM Lite UI is expected to need (primary key, `site_id`, `form_id`, unix-ms `created_at`, optional name / phone / message, required `email`, free-form `extra_fields` JSON for non-canonical fields, `page_path`, `user_agent`, `ip_country` from the Cloudflare edge, an integer `turnstile_pass` flag, a `status` column with the CRM Lite lifecycle enum defaulting to `new`, and `notes`) plus the `leads_site_created` and `leads_status` indexes.
- A `POST /api/forms/contact` endpoint on the public-site Worker that: requires `application/json` content type; rejects malformed JSON bodies; silently drops submissions whose honeypot field is filled (returns a success-shaped 200 with no DB row and no notification email); validates that `email` is present and well-formed; when a Turnstile secret is configured, verifies the supplied `turnstile_token` and rejects the request when verification fails; otherwise generates a unique lead id, reads `CF-IPCountry` from the request, inserts a row with status `'new'`, and returns a 200 JSON response containing the lead id.
- A best-effort Resend email notification to the configured owner address that runs after the lead row is persisted. Resend failures are logged but never fail the request — the lead is the source of truth, the email is courtesy.
- Non-canonical submission fields are not silently lost: any property in the submitted JSON that is not part of the canonical contact field set is stored in the row's `extra_fields` JSON blob.
- When the static site generator renders a page containing a `contact-form` module and a Turnstile site key is configured at build time, the generated HTML includes the Cloudflare Turnstile script tag and a `<meta name="fc-turnstile-site-key">` so the widget can render against the form's `data-turnstile-target` mount.
- The contact-form client island, when it intercepts a form submission and the page has a rendered Turnstile widget, reads the widget's response token and attaches it to the JSON body as `turnstile_token` before posting.

Out of scope (covered elsewhere or deliberately deferred):

- CRM Lite UI for browsing, transitioning, or replying to leads — later capability. Status transitions in this story default to `new`; no transition logic exists yet.
- Lead deduplication, auto-reply to the submitter, magic-link "manage this inquiry" follow-up, and webhook / Slack notifications — future.
- Multi-site lead routing — `site_id` is taken from the Worker's `SITE_ID` env var and is `1stcontact` for now; per-site routing arrives with customer-site support.
- Operator setup steps (D1 database creation, migration apply against the remote DB, Turnstile and Resend account setup, secret `wrangler secret put`) — documented in `apps/public-site/wrangler.toml` comments and the source ticket, but not part of the runtime behaviour this story asserts.
- The static structure of the contact-form module (fields, honeypot input, Turnstile target element) — that surface belongs to the Framework Module Catalog (content modules) story.

## Technical Context

- Resend is invoked only after the lead row is committed, and any error from the Resend POST is caught and logged without rejecting the response. The intent ticket states this explicitly: "Resend failure is logged but does NOT fail the request — the lead is already persisted."
- Honeypot submissions return a JSON-shaped success body but write no row and trigger no email. The intent calls for the honeypot path to be indistinguishable to attackers from a successful submission. The current implementation also returns `dropped: true` in that body; this is an implementation detail beyond the spec but not in conflict with it.
- Turnstile verification is gated on the `TURNSTILE_SECRET` env var being configured. When the secret is absent (typical for local dev without a Turnstile site set up), the handler skips Turnstile verification and persists the lead without a token. When the secret is present, the handler requires a `turnstile_token` field in the submission and rejects the request if either the token is missing or `siteverify` returns `success: false`. Failed Turnstile verification means no DB write and no Resend call — matching the intent ticket's stated behaviour.
- The bundle ticket defines the error code set as `INVALID_JSON | INVALID_CONTENT_TYPE | MISSING_FIELD | INVALID_EMAIL | TURNSTILE_FAILED | INTERNAL`. The handler returns these codes in the JSON error body alongside the HTTP status; ACs reference the codes by name where the intent does.
- The intent ticket lists `name` as a canonical contact field but does not declare it required. The current handler validates only `email`; this matches the intent, which lists `name` as optional in the schema (`name TEXT,` — nullable).
- The Cloudflare Workers Static Assets binding (`ASSETS`) and the marketing-site definition that uses the contact-form module are owned by the Public Site Delivery story (STORY-44). This story assumes that wiring exists and asserts only the form-submission behaviour layered on top of it.
- The contact-form client island falls back to reading a `cf-turnstile-response` hidden input when the global `window.turnstile` API is not available. This is a robustness implementation detail; the AC asserts only the observable contract that a rendered widget's token is attached to the JSON body.
- ULIDs were named in the intent ticket as the lead id format. The current implementation uses `crypto.randomUUID()` instead. This is an internal divergence — the id is opaque to callers; the AC asserts only that the response body contains a non-empty `lead_id` matching the row's primary key.

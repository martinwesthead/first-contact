## Story
**As** the platform operator, **I want** sites, the accounts that own them, and their publish history persisted in a durable multi-site data store with enforced slug uniqueness and a bootstrapped first site, **so that** every site's working draft and live published definition survive restarts, can be rolled back to any prior publish, and can be addressed by a unique, well-formed slug in the shared *.1stcontact.io namespace.

## Description
Establishes the persistent data model behind the platform:

- **accounts** — operator identity (unique email), display name, and a `plan_tier` of `trial` or `paid` used to gate platform actions.
- **sites** — one row per site owned by an account, carrying a globally-unique `slug`, a working `draft_definition`, and a `published_definition` (null until first publish) plus pointers (`published_at`, `published_revision_id`) to the live snapshot.
- **revisions** — one immutable snapshot per publish event, linked to its site, recording the published definition, publish time, and publisher.
- **slug allocation** — rules for what makes a slug well-formed, a reserved-name set that operators may not claim, and collision-recovery suggestions.
- **bootstrap seed** — a one-time seed that brings the platform-owned account and the existing 1st Contact marketing site under D1 management, so the file-backed and D1-backed paths converge.

The data model intentionally supports per-account multi-site even though v1 UX exposes only the single-site path. Migrations are reversible and locally testable.

**In scope**: the three tables, their indexes and constraints, slug validation/reservation/suggestion behavior, and the 1st Contact bootstrap seed.

**Out of scope** (deferred to later REQs): magic-link/session tables, custom-domain mappings, asset (R2) records, audit-log table, and soft-delete columns. v1 sites are not deletable.

## Technical Context
- Relates to CAP-32 (Site Definition Schema): the JSON stored in `draft_definition` / `published_definition` / `revisions.definition` is a Site definition validated by CAP-32's validator. This story owns persistence and slug allocation; CAP-32 owns the in-memory contract.
- **Intent/code divergence to flag for regression**:
  - The REQ-10 spec names migrations `002`/`003`/`004`/`005`; the shipped code uses 4-digit numbering (`0002`–`0005`) to match the existing `0001_create_leads.sql` convention.
  - The spec describes `published_revision_id` as "FK to revisions.id"; the shipped `sites` table declares it as a plain column (soft pointer, no FK) so the seed can set it on the site row before the corresponding revision row is inserted. Only `sites.account_id -> accounts(id)` and `revisions.site_id -> sites(id)` are declared foreign keys.
- **Coordination note (deferred, not in this story)**: REQ-20's `extractAccountId` remains header-driven (`x-account-id` with a `default` fallback); swapping it to session/magic-link resolution is explicitly deferred to the auth REQ and is out of scope here.

## Symptom

While testing the 1stcontact builder chat, the operator repeatedly hits a
"budget / limit" that stops the agent mid-task. No budget was ever requested;
the throttle is self-imposed safety scaffolding that blocks iterative testing.

Two operator asks:
1. A session-reset button at the top of the chat (blank page, new session).
2. The "session budget" raised so high it is never hit during testing.

## Root cause (investigation)

- The browser-rendering `sessionMaxSeconds` budget — the only control literally
  named a "session budget" — is already effectively infinite (1e9s, BUG-17) and
  no caller lowers it. It cannot be the blocker.
- The actual stop is `MAX_TOOL_TURNS = 8` in `apps/control-app/src/chat.ts:130`:
  the per-message tool-use loop hard-caps at 8 turns. A multi-step build
  (analyze_page → transcribe_site → several state edits) exhausts it and the
  agent stops. This is the "safety control around the tools" being hit.
- `apps/control-app/wrangler.toml:57` carries a stale comment referencing a
  "default 50s" per-session budget that no longer exists — confusing noise.

## Fix

1. Builder chat session reset: add a "New session" control to the top of the
   chat panel (`packages/builder-ui/src/components/chat-panel.ts`), wired through
   `main.ts` to create a fresh chat session and reset the in-view history to a
   blank page. Reuse the existing `createSession` / store-reset plumbing.
2. Remove the testing blocker: raise `MAX_TOOL_TURNS` to a value that is never
   hit in interactive testing. Fix the stale `wrangler.toml` budget comment.
   (Browser-render + rate-limit budgets are already effectively unlimited; no
   change needed there.)

## Test plan

- `test_UAT_FC_BUG-19_*` (builder-ui): clicking the new-session control creates
  a fresh session and clears the chat view to blank.
- `test_UAT_FC_BUG-19_*` (control-app chat): a chat exchange that drives >8 tool
  turns runs to completion without being truncated by the turn cap.

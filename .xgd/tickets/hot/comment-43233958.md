Diagnosis for issue 1 (multi-page tabs navigate iframe to control-app root):

**Root cause.** The framework renderer is shared between two emission targets — production Astro static-site output and the in-browser builder preview — but the nav-href format only matches production.

- `packages/framework/src/render/browser.ts:147-152` — `navHref` returns absolute paths like `/menu`, `/contact`. Correct for production (Astro emits separate HTML files per page).
- `packages/builder-ui/src/preview.ts:9-19` — preview iframe is populated via `document.open / write / close`. **No link-click interception**, no client-side page-switching logic.
- `apps/control-app/src/index.ts:63-66` — control-app catch-all returns "Hello from app.1stcontact.io" for any unknown path, so the iframe's nav to `/menu` ends up rendering that.

The preview was built for the single-page case; the multi-page case was never finished. Fix shape: emit fragment hrefs (`#/menu`) from `navHref` when invoked in preview mode + `hashchange` listener on the iframe's content window that re-runs `renderSiteToHtml(site, { pageId })`. Alternative: intercept anchor clicks in the iframe. Fragment shape is preferred — declarative, survives iframe reloads. Production output unchanged via a `target` option defaulting to production. ~3 SP.

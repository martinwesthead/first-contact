## Key technical correction to the suggested fix

The suggested edits assume `AstroContainer.renderToString` emits the scoped
`<style>` block (citing AC-417 as precedent). I verified empirically that it
does **not** — `renderToString` output contains no `<style>` tag, and the
existing AC-417 header test in fact reads its CSS from the module source, not
from rendered output. The rendered output **does** carry the variant class.

I therefore applied the achievable form of the assessor's intent: execute the
real component entry point (the precise defect in AC-431 was that it rendered
nothing) and verify the responsive CSS contract **anchored to the rendered
variant class** — the AC-417 pattern. Each services-grid test now renders the
module, asserts the list element + variant marker are emitted, and proves the
`repeat(N,1fr)` rule lives only inside the `min-width:768px` media query.

## Editor judgment on AC-423 (code-issue NOT escalated)

The editor note raised a possible prod-CSS conformance gap. Investigated:
- header `gap: 4px` (== --space-1) and `border-radius: 999px` (~ --radius-full
  = 9999px) on the hamburger toggle are token-substitutable literals.
- All genuine *content* spacing in the three modules already uses var(--space-*);
  the only literals are mechanical/decorative (hamburger-icon geometry, sr-only
  clip, centering `0 auto`, zero resets, pill radius). The hero `rgba(0,0,0,0.3)`
  is in `text-shadow` (a decorative scrim), not a color/background surface, so it
  is correctly out of scope for the theme-token contract.

I judged these legitimate non-token mechanical values rather than intent
violations, and resolved via documented allowlist (the assessor's primary
recommendation) rather than escalating to code-issue. If the team prefers strict
tokenization, a follow-up upgrade could swap `gap:4px`→var(--space-1) and
`border-radius:999px`→var(--radius-full); the scanner's allowlist comments flag
exactly those two for that decision.

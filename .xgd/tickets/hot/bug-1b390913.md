## Violation

`xgd_source/dashboard/server.py:56` contains `_transcript_direct_append()`, which writes directly to comment ticket `.md` files using `fcntl.LOCK_EX`, bypassing the ticketing API. This violates the rule in CLAUDE.md / FREE-CODING.md:

> Tickets MUST be accessed exclusively through the ticketing API (`xgd ticket` CLI or `xgd_source/core/ticketing` module). NEVER read, write, or modify ticket `.md` files directly via filesystem operations.

Direct file access bypasses locking, validation, indexing, and the API contract that all other code relies on.

## Why it exists (rationale from server.py:684-689)

> The cold index rebuild can hold LOCK_EX for 70+ seconds (22K+ file scan under I/O pressure). To avoid the LOCK_EX cost of ticketing.update(), we use `_transcript_direct_append()` for existing comments — per-file fcntl, no index lock. ticketing.comment_create() (new session) still needs LOCK_EX but is a much rarer event (once per chat, not once per message).

So the bypass is a workaround for a perf problem in the cold-index LOCK_EX path. Per-message chat appends were paying a 70-second hang during cold-index rebuilds.

## Resolution paths (pick one)

1. **Fix the cold-index LOCK_EX-for-70s root cause.** Make rebuild non-blocking, acquire LOCK_EX only for the small final swap, or rebuild incrementally. Then the bypass is unnecessary and can be removed without further work.
2. **Add a first-class API op for per-message append.** `ticketing.comment_append(uid, kind, text)` that locks only the comment file (not the index) and queues an index update. Dashboard calls that instead of bypassing.
3. **Defer to the substrate refactor.** A future substrate (per CHAT-22 discussion in first-contact) would replace this layer entirely. The bypass becomes a non-goal in the new design.

## Acceptance criteria

- `_transcript_direct_append()` is removed.
- All chat-transcript append paths go through the ticketing API.
- Per-message append latency under normal load remains acceptable (no 70s hangs during cold-index rebuild).

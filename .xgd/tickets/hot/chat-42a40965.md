Given the Framework chat's commitment to **fixed section library + theme tokens + content JSON**, the in-browser render question gets a clean answer.

## Why in-browser rendering is borderline free here

The framework design has already made the hard call: the renderer is "given a site definition (theme tokens + page composition + content JSON), produce DOM using a shared section library." This is *exactly the same code* the static generator would run at build time. So:

- **The renderer is a JS module** that imports the section library, takes a JSON definition, writes DOM. Same code runs in the static generator (build to disk) and the builder (render live).
- **Theme changes are CSS variables flipping** — sub-frame.
- **Content edits are JSON state changes** — re-render the affected section.
- **Section reorder** is array reorder — re-render the page.

No build step, no network round-trip. For ~10 sections, render time is well under a frame.

## The complexity, honestly

Three real concerns, none deal-breakers:

1. **CSS / style isolation.** The builder UI has its own CSS (chat panel, splitter, tooling). The preview needs its own style scope so theme tokens don't leak into the chrome. Standard solution: render into an `<iframe srcdoc>` or same-origin `<iframe>` you control. You can call `iframe.contentDocument` directly — no postMessage gymnastics. Full CSS isolation without a build server. ~20 lines of plumbing.

2. **Section-library distribution.** Section library has to be shippable as importable ESM modules — both the static generator and the builder app consume it. If the static generator was already going to be a Node/Worker program that imports those same modules, no-op. If it was going to be a separate templating system, you now have to converge them. The framework chat's design naturally converges them; don't let them diverge.

3. **Site definition needs a strict, versioned JSON schema.** The AI (writing diffs), builder UI (applying them), and static generator (rendering for production) all read the same shape. Schema drift = bugs. Pin it down early; validate at every boundary.

## Where iframe-of-built-site is still useful

Not as the main loop — as a **"verify what ships"** sanity check. One button: "open as the real built site." A Worker runs the actual static generator and serves from a preview URL. Slow (seconds), used rarely. Catches drift between live renderer and production build. Keep it; don't put it on the critical edit path.

## Comparison to iframe-rebuild-on-every-change

Buys you nothing in this architecture (the live renderer IS the production renderer) and costs a round-trip per chat turn. The Durable Objects "live preview sessions" line in DOC-5 is better repurposed for **session state** (multi-user, chat history, undo stack) than build queueing.

## Net recommendation

- **Main loop:** in-browser render into a sandboxed same-origin iframe. State lives in the builder app; iframe is a dumb renderer.
- **Pipeline:** `siteDefinition (JSON) → sectionLibrary[type].render(props, themeTokens) → DOM`. Same module path used by the static generator.
- **Schema:** lock JSON shape early; validate at AI output, builder state, generator input.
- **Production-fidelity check:** "view as built" button hitting a Worker preview build. Optional, off the critical path.

This also feeds back into CHAT-7: the framework's "structured changes only, AI never writes CSS" rule is what makes in-browser render trivial. If the framework loosens that — letting AI inject custom CSS or HTML — the in-browser path gets messier (you'd need a CSS sanitizer/scoper) and iframe-build starts looking better. So the two chats reinforce each other: **lock structured-only AI edits → in-browser render is the right call**.

Where next — site-definition JSON schema sketch, or the chat → diff → render protocol shape?

## Why mechanical validation is the linchpin

The whole architecture pivots on "AI emits structured edits." That bet only pays off if **invalid edits are mechanically impossible to ship**. Without a hard validation gate:

- A bad dial value escapes to D1 and crashes render
- An `asset-ref` to a deleted image silently breaks layout
- AI mishearing a dial enum (`shape: 'cirle'`) corrupts state
- Framework upgrades quietly invalidate older stored definitions
- A clever content field smuggles HTML/CSS past §6.3's forbidden list

Validation is what makes "structured changes only" a real guarantee instead of an aspiration.

## What mechanical validation can actually check

Each of these is deterministic, sub-100ms, no DB lookups:

1. **Schema conformance** — the site definition matches `packages/site-schema`. JSON Schema or Zod. Required fields present, types correct, no unknown keys.
2. **Module-meta conformance** — for each module instance: `type` exists at `version`; `variant` ∈ `moduleMeta.variants`; every dial value ∈ `moduleMeta.dials[name]`; content fields match `moduleMeta.contentSchema`.
3. **Theme-token conformance** — every set token is in the framework's contract; values pass type checks (color is valid hex/named, scale step is in range).
4. **Referential integrity** — every `asset-ref` resolves; every nav entry points at a real page or anchor; every module ID referenced in nav exists.
5. **Uniqueness / structure** — module IDs unique within a page; page slugs unique within a site; nav entries non-circular.
6. **Content-field safety** — markdown fields don't contain inline `<style>`/`<script>`/raw HTML beyond what the renderer allows; URL fields are well-formed.
7. **Render smoke** — `tools/generate` can render without throwing. A fast no-output dry run, optional, expensive.

## Where the gate sits — four layers, same validator

| Layer | What runs | On fail |
|---|---|---|
| **AI tool call** | Each structured edit validated against current state + framework catalog *before* applying | Reject; return machine-readable error to AI; AI self-corrects on retry |
| **Builder state** | Builder UI refuses to write an invalid definition into local state | Toast + revert; never silently corrupted |
| **Save to D1** | `control-app` Worker re-validates server-side before persisting | 4xx; client surfaces error; defense in depth — client could be compromised |
| **Build / publish** | `tools/generate` validates before render | Block publish; clear error to operator; only happens on framework drift |

The actual validator is one function: `validate(siteDefinition, frameworkCatalog) → Result`. It lives in `packages/site-schema` and is consumed by everything. Same rules, same code, every layer. Drift between layers is the only way invalid data ships — collapse that drift by having one validator.

## Cost worth naming

The AI-tool-call gate has a retry cost: invalid edit → AI re-prompted with the error → second API call. Mitigations:

- The schema goes into the AI's system prompt so most edits are valid first try
- Validator errors are *machine-readable* (`{ path: 'modules[2].dials.shape', expected: ['square','rounded','circle'], got: 'cirle' }`) so the AI fixes precisely, not by guessing
- For common errors, schema descriptions are tight enough the AI doesn't try (e.g., describe `shape` as `'square' | 'rounded' | 'circle'` literally, not as "a shape value")

Empirically, with a well-described schema, retry rates should be low single digits.

## What this means for the doc question

This is the missing piece that tips it. Mechanical validation is:

- A **framework guarantee** — part of "structured changes only" actually being safe. Belongs in DOC-7.
- A **builder principle** — the chat loop's feedback mechanism depends on it. Belongs in DOC-8.

It's the right kind of cross-cutting thing to make explicit on both sides. Concretely:

- **DOC-7** gets a new section (suggest §6.5 or new §7): "Mechanical validation is a framework guarantee. The framework provides a single validator over `(siteDefinition, catalogVersion) → Result`. No code path may write to D1 or render without it." Lists the validation categories above.
- **DOC-8** describes the AI-loop side: tool calls validate before state apply; AI receives structured errors; retry contract.

I'd still build a small DOC-8 rather than fold everything into DOC-7 — but DOC-7 has to make the validator a first-class framework commitment, because every consumer (builder, generator, future API clients) depends on the *same* validator existing. If the validator is just "a thing the builder does," it drifts.

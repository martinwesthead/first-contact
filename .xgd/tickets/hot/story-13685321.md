## Story
As an operator, I want to upload, list, retrieve, and delete site assets through HTTP routes backed by R2 storage, so that downstream features (screenshot persistence, asset editing, customer-site asset references) have a stable storage surface to depend on.

## Description
Provides a single shared asset bucket exposed through four HTTP routes on the control-app worker. Assets are addressed by string key. Each upload writes (or overwrites) bytes alongside a caller-supplied content type; each retrieval returns those bytes with the same content type; the list endpoint enumerates current objects with size, etag, upload timestamp, and content type; the delete endpoint removes an object. Concurrent overwrites are guarded by an optional `If-Match` etag precondition that rejects stale writes with `412`.

In scope:
- Four HTTP routes: list, get, put, delete.
- Optional `If-Match` etag concurrency on put.
- Local-emulation parity (`wrangler dev` / Miniflare) so the full round-trip exercises without provisioning a real bucket.

Out of scope:
- Per-customer-site asset isolation — single shared bucket in v1; revisit when auth lands.
- Authentication on the routes — unauthenticated stub matching the rest of the control-app in v1.
- Soft-blocking deletes against published-revision references — that policy belongs to the publish/revisions intent.
- The assets-tab editor UI that consumes these routes (separate intent).

## Technical Context
- Implements the R2-assets half of intent [[bundle-bbb1bd9c]] (REQ-20 acceptance criteria 13–17); the safety-layer half of the same intent is documented in a separate story.
- Bound under `ASSETS_BUCKET` in `apps/control-app/wrangler.toml` (root environment) and under `[env.production]`; bucket name `1stcontact-assets`.
- Production-grade authentication intentionally deferred; routes trust the network in v1.
- Code reality not explicitly stated in intent (worth surfacing during regression so divergences are visible rather than silently absorbed):
  - The list endpoint accepts optional `prefix` and `cursor` query parameters and returns a `{items, truncated, cursor}` envelope rather than a bare array.
  - Keys containing `..` segments or starting with `/` are rejected with `400` — a sensible safety extension not specified in intent.
  - The retrieval route supports `HEAD` as well as `GET`, returning content-type / etag / content-length headers with no body.
  - A put without a request body returns `400`.
- Downstream consumers documented in intent: screenshot persistence (reference-digest, browser-rendering) and the assets-tab editor UI — each implemented under their own intents.

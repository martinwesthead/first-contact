## Story
**As an** operator building a site in 1st Contact, **I want** to paste the URL of an
existing website and have it converted — immediately, with no confirmation prompt —
into a freshly-cleared editable draft of my own site, **so that** I can reproduce a
recognizable copy of that site within the framework instead of building it from scratch.

## Description
This story covers the **mechanical convert-flow orchestration** that turns a
previously-analyzed Reference Digest into a draft the operator can iterate on.

In scope:
- The operator-triggered convert action. Conversion proceeds immediately on every
  invocation under any starting condition — there is no destructive-overwrite
  confirmation gate, no companion confirmation action, and no per-URL/per-session
  consent. (The confirmation gate and its "I own this site" robots piggyback were
  removed; the operator's safety net is the Reset button plus the unconditional
  clear-to-scaffold step below, backed by git/publish history.)
- An unconditional **Stage 0 clear-to-empty-scaffold**: before any mechanical work,
  the convert clears the operator's draft to a minimal 1-page empty scaffold (one
  home page at slug `/`, framework-default theme tokens, `in-page-anchors` nav with
  no entries, no modules, `config.businessName` seeded from the source site title or
  `"Untitled"`). Clearing is unconditional — it runs even if the draft is already
  empty — so the AI reconstruction lands on a clean slate with no prior-draft or
  1stcontact-starter residue. A stage 0 `cleared` progress notification is emitted
  and the cleared scaffold definition is returned so the builder applies it to the
  working draft before any subsequent edits.
- Staged progressive-reveal progress notifications during a successful conversion
  (screenshot → theme tokens → digest written) streamed to the builder.
- Producing and persisting a per-site TranscriptionDigest artifact (theme tokens,
  per-page plan, R2-mirrored asset inventory, mirror summary) and returning a
  mechanical completion result `{ kind, digestKey, clearedSiteDefinition, summary{
  pageCount, assetCount, mirrored, mirrorFailures } }` — the only site object in the
  payload is the empty cleared scaffold; never a source-synthesized site.
- Wiring the reproduce-a-website how-to guidance into the builder AI's system
  prompt so the chat AI acts as the synthesizer (reads the digest, then
  reconstructs the draft via its structured-edit and page-CRUD tools).
- End-to-end coverage of the single-page "killer demo" and the multi-page flow.

Out of scope (covered by sibling stories in this capability):
- TranscriptionDigest derivation primitives and the digest read-back action.
- Asset mirroring to R2.
- The page-CRUD structured-edit tools.
- The TranscribeProgress chat-card variant (including its Stage 0 row).

This flow is purely mechanical: there is no internal LLM site-synthesis step and
no fallback path to legacy synthesis behavior (per CLAUDE.md "No Legacy Modes").
The chat AI is the synthesizer.

## Technical Context
Reconciled from BUNDLE-4: REQ-28 (Site Transcription Layer B) **as reshaped by
REQ-30** (mechanical transcribe_site + R2 digest + AI-driven reconstruction), then
upgraded by BUNDLE-5 (REQ-35 + REQ-34).

REQ-35/REQ-34 net effect on this orchestration:
- The destructive-overwrite confirmation gate is removed: no `requires_confirmation`
  early return, no `convertConfirmed` consent storage, no `confirm_convert` operator
  action, and no convert-path "I own this site" robots-override registration. The
  underlying per-origin `robotsOverrides` storage from REQ-20 is retained, but no
  convert path writes to it (only a read-side `.has()` remains).
- A Stage 0 clear-to-empty-scaffold runs on entry, before mirror/digest. The empty
  scaffold is built by `buildEmptyScaffold({ businessName })` where `businessName`
  comes from `titleFromDigest(homeDigest)` (H1 → slug-derived title → hostname →
  `"Untitled"`).

Architecture: Cloudflare Workers operator API; ASSETS_BUCKET (R2) holds the digest
artifact; FETCH_CACHE_KV holds analyzed Reference Digests.

Implementation deviations from the intent body, surfaced for regression:
- **Clear is FE-applied, not server-persisted.** REQ-34's intent text says the clear
  writes through "the same persistence path used by state_edit actions". The code
  does NOT mutate a server-side working draft; it builds `clearedSiteDefinition` and
  returns it in the completion payload, and the builder applies it to its working
  store. There is no server-side `workingSite` mutation in the handler.
- **Digest lookup source.** The intent text says "fetch the digest from chat
  history". REQ-23/REQ-24 (persistent chat) are deferred, so the action reads the
  analyzed digest from the `analyze_page` KV digest cache keyed by
  `sha256(url|schemaVersion)`. The `digestId` input is the source URL the operator
  pasted; conversion of a never-analyzed URL fails with `digest_not_found`.
- **Synchronous staging.** Stages run sync-await inline within the action. The
  emitted stage numbering is an artifact of the REQ-28→REQ-30 reshape plus REQ-34's
  prepend (clear=0, screenshot=1, theme=2, asset events=4, body-copy capture=5,
  final digest=3) and does not reflect a strictly monotonic ordering.
- **siteId == accountId.** The digest is written to `sites/{accountId}/transcription/
  digest.json`; current code uses the account id as the site id.
- **LLM mocked in tests.** The end-to-end demos drive a scripted (mocked) chat loop;
  real-Claude conversion is a manual smoke test, not a gated test.
- **Per-module confidence dropped** (REQ-30 OUT). Confidence is implicit in the AI
  narrative; only a theme-token confidence band is emitted at the theme stage.

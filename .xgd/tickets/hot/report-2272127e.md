### Why this overlap is acceptable

The survey flagged the boundary because both stories materialize a `Site` definition into rendered HTML. They share the **framework module catalog** (CAP-34) — modules are registered once and consumed by both render entry points — but each story owns a distinct render path with a distinct consumer:

- **Builder UI (CAP-38)** consumes the catalog browser-side via `@1stcontact/framework/render` to populate the preview iframe on every accepted edit. No file I/O, no Node modules. Storage is localStorage.
- **Static Site Generator (CAP-35)** consumes the catalog Node-side via Astro Container to emit a deployable static bundle. File I/O is the whole point. Storage is the output directory.

This is shared dependency, not shared capability. The catalog dependency is correctly documented on STORY-46 ("Dependencies → CAP-34 / Framework Module Catalog") and is implicit on STORY-43 (the framework registry is its render path). No code is duplicated between the two stories — they consume a third capability — and the user-visible behaviours are disjoint (interactive live preview vs. deploy-time static build). The cluster is a domain-neighbour overlap, not a scope overlap; the boundary holds.

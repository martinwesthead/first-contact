## 1. Running locally

The Cloudflare-native stack has a clean local story: **`wrangler dev`** boots the Worker API plus serves the static assets, and `wrangler` provides local emulators for D1 (SQLite-backed), R2 (filesystem-backed), KV, and Durable Objects via bindings declared in `wrangler.toml`. One `npm run dev` ties it together. Hot reload on the Worker is built in; for the builder SPA we'd want Vite (or similar) running alongside, proxied through the Worker for same-origin behavior.

The one decision worth making early is **monorepo shape**: a single repo with `apps/control` (builder SPA + portal), `apps/api` (Worker), `packages/framework` (section library + renderer + token system), `packages/site-schema`, and `sites/first-contact/` (the FC marketing site definition) is the smallest layout that supports the hybrid model below without circular dependencies.

## 2. Auto-publish on xgd-stable

Straightforward: a GitHub Action on `push: branches: [xgd-stable]` that runs `wrangler deploy` with a Cloudflare API token in repo secrets. Two deploys per push:
- the Worker (API + control app's static assets)
- the FC marketing site (static output → Workers Static Assets, separate Worker or a route prefix)

Worth deciding now: **one Cloudflare account, one project, two Worker services** (api + public-site), or **one Worker** with route-based dispatching. I'd lean two Workers — cleaner separation, independent rollback, the public site stays up if the API is being rebuilt.

## 3. The hybrid — FC built as an FC site

Yes, this maps cleanly onto DOC-5's existing split. There are really two surfaces:

| Surface | Built how |
|---|---|
| **Control app** (builder UI, portal, dashboard) — authenticated, stateful, interactive | Custom code (SPA + Worker APIs) |
| **Everything else on first-contact.com** (homepage, pricing, about, blog, docs) | An FC site, built by the FC framework, dogfooded |

The CHAT-7 Phase 0 plan (single-page site, ~6–8 modules, GenDev Labs as the first site) **is** this hybrid — extended to mean the FC marketing site is also the canonical demo and gets republished automatically on every xgd-stable advance. The chicken-and-egg is real but small: framework Phase 0 has to land before you can build the FC marketing site as an FC site, so the very first publish from xgd-stable would just be the framework + an empty placeholder. Could hand-roll a temporary homepage if you want something on the air sooner.

**Why two Workers, not one:**
- `public-site` is reachable from anywhere (customer-site form submits, Stripe webhooks) and needs to stay up if the control app is mid-deploy.
- `control-app` is auth-gated and SPA-facing.
- Independent deploys, independent rollback, shared D1/R2 bindings.

**Why pnpm:** best workspace ergonomics; smaller node_modules; deterministic; cleaner than npm workspaces for this shape. Open to npm if you'd rather not adopt a new tool.

## Local dev

Root scripts:
```
pnpm dev          # boots wrangler dev for both apps + builder-ui vite + static-gen watcher
pnpm dev:public   # just public-site
```

Each app's `wrangler.toml` declares bindings to **local** D1 (SQLite under `.wrangler/state/`), R2 (filesystem), and KV. Same binding names as production; no code differences between local and prod.

```yaml
  push:
    branches: [xgd-stable]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - uses: actions/setup-node@v4

      - run: pnpm install --frozen-lockfile
      - run: pnpm -r build
      - run: pnpm generate                # sites/first-contact → public-site/public/
      - run: pnpm wrangler d1 migrations apply first-contact --remote
```

## Where to find Cloudflare credentials

**`CLOUDFLARE_ACCOUNT_ID`** — quickest path:
- `dash.cloudflare.com` → click into **Workers & Pages** (left sidebar) → the Account ID is in the right-hand sidebar.
- Also visible at the bottom of any zone's overview page (e.g. clicking into `1stcontact.io`).

**`CLOUDFLARE_API_TOKEN`** — generate one:
1. `dash.cloudflare.com/profile/api-tokens` (or User icon → My Profile → API Tokens)
2. **Create Token** → start from the **"Edit Cloudflare Workers"** template (covers most of what you need)
3. Then **add** these scopes for the full platform:
   - Account → **Workers Scripts** → Edit *(in the template)*
   - Account → **Workers KV Storage** → Edit *(in the template)*
   - Account → **Workers R2 Storage** → Edit
   - Account → **D1** → Edit
   - Zone → **Workers Routes** → Edit
4. **Account Resources**: include the account holding 1stcontact.io
5. **Zone Resources**: include the `1stcontact.io` zone
6. TTL: leave indefinite (or set a renewal calendar reminder)
7. Continue → Create → **copy the token immediately** (it's shown only once)
